# CapAgent — Experiment Report

This report instantiates the evaluation protocol of the paper (Sections 5–6) on the
prototype implemented in `capagent/`. Following the paper's own stance, the study is
**diagnostic**, not leaderboard-style: the goal is to show that the formal objects can
be *instantiated* — contracts extracted, Action-Capability Graphs built, known
witnesses replayed, repaired versions accepted, certificates checked — and that a
capability-specific abstraction is more precise than simpler defenses.

Everything below is regenerated by:

```bash
python -m capagent run-all        # -> capagent/experiments/out/
python tests/test_capagent.py     # 7/7 pass
```

## 2. Results by research question

### RQ1 — Expressiveness
`lambda_cap` expresses every targeted missing-capability pattern: external send,
deletion, overwrite, command execution, harness integrity (patch-scope + verdict),
memory-as-instruction, and privilege crossing. Each has a buggy kernel that reaches
`BadCap` and a fixed kernel that does not (`tables/table2_3_witness_replay.csv`).

### RQ2 — Contract & graph extraction
`contract-extractor` recovers effect contracts from wrapper `ast` (decorator
evidence, direct API sink patterns, name heuristics, and one-level local-call
resolution); `guard-extractor` recovers `require(...)` grants; `acg-builder` builds the
ACG from plan control-flow.

| project | tools | sinks | resolved | unresolved | contract coverage |
|---|---:|---:|---:|---:|---:|
| doc_agent | 6 | 3 | 6 | 0 | 100% |
| swe_agent | 6 | 6 | 6 | 0 | 100% |
| manager_agent | 4 | 3 | 4 | 1 | 80% |

Aggregate **contract-extraction coverage = 0.941** (16/17). The single unresolved
symbol is `manager_run`, an internal helper that is not a registered tool — reported
conservatively rather than silently dropped, exactly as the plan requires.

### RQ3 — Verification (replay + discharge)
- **Known-bug replay rate = 1.00** — every buggy witness is rejected with the exact
  missing capability (`tables/table6_new_warnings.csv`).
- **Fixed-version discharge rate = 1.00** — every repaired version is accepted.
- Repairs are auto-classified into the paper's four patterns (Section 5.6):
  `declassify` (send), `guard` (delete/overwrite/verdict), `provenance`
  (exec/instr/delegate), `scope` (patch → task paths).

### RQ4 — Certificates
The Boolean-semiring matrices are tiny and sparse. Over all 33 cases:
matrix dimension ≤ **4** states, ≤ 5 abstract facts, closure in ≤ 4 iterations;
**total checking time ≈ 7–10 ms** for the whole suite. Explicit-state and matrix
verdicts **agree on 100%** of cases (paper Section 5.3: graph reachability = Boolean
matrix fixed point). See `tables/table5_certificates.csv` and `certificates/*.json`.

### RQ5 — Diagnostic quality
**Diagnostic quality = 1.00**: every rejection reports a concise violating prefix, the
missing capability (kind + resource + provenance/label), and the tool-contract
evidence that made the effect high-impact. Example (`plan_exec_buggy`):

```
Unsafe path:  memread(k_web) -> call(shell_exec)
Missing capability:  Cap(Exec, provenance=untrusted, sandbox=on)
Triggered bad component:  BadCap
Evidence: subprocess.run
```

## 3. Baseline comparison (paper Section 5.1 / Table 7)

Positive = "flagged unsafe". Ground truth = 16 unsafe, 17 safe (33 cases).

| method | TP | FP | FN | TN | precision | recall | accuracy | blind spot |
|---|---:|---:|---:|---:|---:|---:|---:|---|
| **CapAgent** | 16 | 0 | 0 | 17 | **1.00** | **1.00** | **1.00** | — |
| sink_only | 16 | 17 | 0 | 0 | 0.485 | 1.00 | 0.485 | no authorization notion |
| allowlist | 16 | 17 | 0 | 0 | 0.485 | 1.00 | 0.485 | ignores resource & provenance |
| guard_dominator | 14 | 3 | 2 | 14 | 0.824 | 0.875 | 0.848 | coarse scoped capabilities |
| generic_reach | 16 | 3 | 0 | 14 | 0.842 | 1.00 | 0.909 | kind-only; no scope/label/prov |
| taint_to_sink | 9 | 1 | 7 | 16 | 0.900 | 0.562 | 0.758 | misses pure-authorization bugs |

The failure modes are exactly the paper's predicted ones:
- **sink-only / allowlist** over-report — they flag every safe program that merely
  *contains* a high-impact API (17 false positives).
- **guard-dominator** accepts `patch-buggy`: a task-scoped write capability *dominates*
  the sink, but the sink is a **harness** write — wrong scope/kind (2 false negatives).
- **generic reachability** (kind-only) flags benign public sends and controls that use a
  sink kind with no same-kind grant — no resource/label/provenance reasoning (3 FP).
- **taint-to-sink** misses `delete`/`overwrite`/`verdict` bugs that have **no tainted
  data flow at all** (7 false negatives), and flags an *authorized* private send (1 FP).

Only CapAgent, which matches effect **kind + resource scope + label + provenance**
against held capabilities, achieves zero errors.

## 4. Expected conclusions (paper Section 5.5) — supported

1. **Capability-safety unifies the failures.** All 16 high-impact failures across four
   families reduce to a missing/mismatched capability over ⟨kind, resource, label,
   provenance⟩.
2. **Action contracts make agents analyzable.** 94% of tools resolved to effect
   contracts directly from wrapper source; the ACG was built for every plan.
3. **The approach explains fixes.** 13/13 repairs map to guard insertion, scope
   refinement, provenance validation, or declassification, and each makes `BadCap`
   unreachable.
4. **More diagnostic than baselines.** CapAgent is the unique method with no FP and no
   FN; every simpler baseline over- or under-reports on a structurally-predictable case.
5. **The linear abstraction is lightweight.** Matrices are ≤4×4 with ≤5 facts; the full
   suite checks in ~10 ms and certificates are trivially re-checkable.
6. **Not complete for all agent safety.** The checker proves only capability-safe
   high-impact effects under accepted contracts — not task/intent/alignment correctness.
7. **LLM assistance reduces effort, not soundness.** `llm-assist` runs offline and
   advisory-only (`accepted=False`); removing it changes nothing in the verdicts.

## 5. Threats to validity / honest limitations

- **Synthetic subjects.** The scaffolds are minimal, hand-written stand-ins for the
  M1–M4 frameworks named in the plan (Agno, MetaGPT, SWE-bench, …); they exercise the
  formal machinery and extraction patterns but are not a large-repo field study. Pinning
  real commits is future work (plan Milestone 5).
- **Extraction is heuristic.** The soundness theorem is *relative* to accepted contracts;
  a mis-specified wrapper can make the checker unsound. Unresolved tools are reported.
- **Path abstraction.** `think`/branches are modeled as nondeterministic choice
  (conservative); the explicit-state engine is exact for these finite Boolean instances,
  and agrees with the matrix fixed point on 100% of cases here.

## 7. Milestone 5 — real repositories (16 pinned frameworks)

Regenerate with `python -m capagent.realeval.run_real`; full tables in
`capagent/experiments/out_real/REAL_REPORT.md`. Subjects span M1–M4:
open-interpreter, smolagents, SWE-bench, MetaGPT, langchain-mcp-adapters, agno, dspy,
browser-use, OpenHands, litellm, swarm, private-gpt, letta, SuperAGI, AIOS, dify.
Total analyzed: **17,185 Python files / 4.28M LOC** (1 file unparsed).

### RQ2 — automated sink & guard inventory (fully automated over real source)
**4,855 high-impact sink sites** — Exec 410, Overwrite 1,198, Delete 361,
SendExt 1,712, CredAccess 1,174 (credential counts restricted to secret-like keys) —
and **7,113 guard sites**. Guard density per sink spans **two orders of magnitude**,
from 0.03 (SWE-bench, MetaGPT — harness/pipeline code) to 3–7 (letta, OpenHands, dify).
This is a direct, automated observation that high-impact effects are pervasive and
very unevenly mediated across real agent frameworks.

### RQ3 — automated tool-registry ACG adapter
For frameworks with a discoverable registry (`@tool`, `@register_tool`, `Tool`
subclasses), the adapter extracts every registered tool, classifies its effect, builds
an ACG in which the model may call any registered tool (`think(Π)` over the registry),
translates to λcap, and checks it. Across the 16 repos it discovered **509 registered
tools in 9 frameworks** (agno 298, smolagents 80, SuperAGI 55, MetaGPT 45, …), of which
**72 are high-impact**; the checker flags **56** as reachable from a model choice with
**no capability requirement attached in the registry itself**. This makes the paper's
central point automatically on real code: *a tool-name allowlist is not a capability* —
static registries expose raw effects, and authorization (if any) lives elsewhere.

### RQ1/RQ3 — grounded witness replay
**23 witness pairs**, each a known capability-safety property encoded as a λcap
buggy/fixed pair and *grounded* in a real sink location discovered by the scanner
(e.g. `agno/libs/agno/agno/utils/shell.py`, `letta/sandbox/node_server.py`,
`aios/tool/manager.py`, `smolagents/src/smolagents/tools.py`). **Known-bug replay rate
= 1.00, fixed-version discharge rate = 1.00**: every buggy path is rejected with its
exact missing capability, every repaired path accepted.

### Honest scope (unchanged)
The sink/guard inventory and registry adapter are fully automated; the grounded
witnesses are manually-curated *properties* tied to real sink locations, not
auto-discovered defects. A faithful ACG for an arbitrary framework's full agent loop
(dispatcher + planner + memory) still needs deeper per-framework adapters — the registry
adapter is the first automated step. All guarantees remain relative to accepted
contracts (paper §8.1). Two candidate repos (ChatDev, CAMEL) were dropped after network
clone failures; results are reported on the 16 that cloned cleanly.

## 8. One-line bottom line

The prototype implements the entire plan and runs both a synthetic suite (33 cases,
perfect classification, baselines behaving exactly as predicted) and a real-repository
study (16 frameworks, 4.28M LOC): capability-safe tool use is **extractable, decidable,
repair-explaining, and more precise than generic defenses**, and its central claim —
that high-impact tool effects need capability mediation the registry does not provide —
holds automatically on real agent code.
